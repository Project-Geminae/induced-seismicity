1. The question we're trying to answer

A regulator looking at an induced seismic event in the Midland Basin wants a number that helps them act. The interesting actions are roughly:

  1. Deny new permits in problematic geology
  2. Cap volumes on existing wells
  3. Order shut-ins after a specific event
  4. Set risk-based monitoring thresholds

The dashboard's per-event interaction (click an event → see contributing wells) is targeted at action #3: "given that this M3.4 just happened, which of these 12 wells are most likely 'responsible' and which are bystanders?"

This is a different question from "what's the average effect of injection on seismicity in the basin?" — that's the population-level ATE that the TMLE shift / dose-response / mediation drivers compute. This page is about the per-event attribution layer that sits on top of the same Q model.

2. Causal estimands and which one we compute

There is no single "the causal question" — there is a hierarchy, and different estimands answer different policy questions:

Estimand Question it answers Have we computed it?
ATE (Average Treatment Effect) "On average across all well-days, what's the effect of injection on expected ML?" Yes — see tmle_shift_*.csv, tmle_dose_response_*.csv
CATE (Conditional ATE) "What's the effect for wells with this depth/formation/fault profile?" Partial — features are in the panel but not yet sliced
ITE (Individual Treatment Effect) "For this well on this day, what's the effect?" Not identified from observational data alone
g-computation prediction "What does the model predict for this well's contribution under its actual injection vs. zero injection, holding all other features constant?" This is what the per-well attribution sidebar shows.
PN (Probability of Necessity, Pearl 2000) "Given that this event happened with this well injecting, what's the probability it wouldn't have happened with the well at zero?" A simplified fractional-reduction proxy is shown; not the formal Pearl PN
Shapley attribution "What's the unique fair share of credit for each well, given non-additive interactions in the Q model?" Not computed; deferred
TMLE is a method, not an estimand. The TMLE machinery can target any of these — what differs is the unit of analysis, the DAG, and what counterfactual you're contrasting against. The per-event attribution shown in the sidebar is a per-well g-computation prediction, not an identified causal effect.

3. What the dashboard actually computes

For each event-radius combination, the pipeline does the following:

3.1. Q model fitting (offline, per radius)

A hurdle Super Learner is fit on the well-day panel (panel_with_faults_<R>km.csv): 

Outcome:    outcome_max_ML  (max ML within R km on this day, or 0)
Treatment:  cum_vol_365d_BBL
Mediator:   bhp_vw_avg_365d  (depth-corrected BHP from volume-weighted WHP)
Confounders: perf_depth_ft, days_active, Nearest Fault Dist (km),
             Fault Segments ≤ R km, formation (one-hot top-K + OTHER)

Q model:    Hurdle Super Learner
            Stage 1: P(Y > 0 | features)              logistic stack
            Stage 2: E[log(1 + Y) | Y > 0, features]   regression stack
            Combined: Q(features) = P_pos(features) · expm1(stage_2(features))

            Base learners: RidgeCV + XGBoost (GBM dropped for runtime)
            Meta-learner:  RidgeCV non-negative ensemble

This Q model is conceptually different from the Q used by the TMLE shift / dose-response / mediation drivers. Those are fit at the cluster-day level (multiple wells aggregated into one row before the model sees them) and are used to estimate population-level estimands. The dashboard Q is fit at the well-day level so it can be applied to individual wells.

Code: build_attribution_q.py. Output: q_attribution_<R>km.pkl for R = 1..20 km.

3.2. Per-event attribution (live, on click)

When the user clicks an event:

  1. For each well within R km of the event, build its feature row: (cum_vol_365d, bhp_vw_avg_365d, perf_depth_ft, days_active, fault_dist, fault_count, formation_one_hot)
  2. Compute Q_factual = Q(actual features)
  3. Compute Q_counterfactual = Q(features with cum_vol set to 0 and bhp set to depth-only hydrostatic baseline 0.45 psi/ft × perf_depth_ft)
  4. contribution = Q_factual − Q_counterfactual
  5. share = max(contribution, 0) / Σⱼ max(contributionⱼ, 0)
  6. PN_proxy = max(0, min(1, 1 − Q_counterfactual / Q_factual))

3.3. Why this counterfactual

The shut-off counterfactual zeroes the cumulative-volume treatment AND replaces the BHP feature with the value the well would have had at zero wellhead pressure (just hydrostatic from the brine column at the perforation depth). This keeps the counterfactual on the support of the training distribution: a well that has perforations at 9000 ft but isn't injecting still has ~4050 psi of bottom-hole pressure from the column weight alone. Setting BHP to literally zero would push the input outside the support and produce nonsense extrapolations.

4. Assumptions and disclaimers

The contribution numbers are model predictions, not identified causal effects. They are conditional expectations from a Q model fit on observational data, applied to factual and counterfactual feature vectors. They are useful for ranking wells and for "back of the envelope" attribution, but they are not the same thing as the well-day potential outcome under intervention.

The full set of assumptions:

  • No unmeasured confounding. The DAG assumes that (cum_vol, BHP) capture all the injection-side variation that affects nearby seismicity, and that the listed L variables capture all the confounders. Pre-existing pore pressure, fluid chemistry, fault stress state, and basement coupling depth — not in the panel.
  • Positivity / overlap. The "shut off" counterfactual should be inside the support of the training data. Wells that are injecting at extreme volumes and in unusual formations may have counterfactual feature vectors that the Q has never seen. This is part of why the simple cum_vol → 0 counterfactual was too aggressive without the BHP fix.
  • The Q is honest in the population sense, not the per-well sense. The Super Learner stack was fit to minimize cross- validated MSE across ~600k well-day cells. Individual cell predictions are unbiased on average across the training distribution, but the per-cell error variance is large. A single click's per-well numbers are noisy.
  • The PN proxy is not Pearl's formal PN. Pearl & Tian (2000) define PN = P(Y_{do(X=0)} = 0 | Y_{X=actual} = 1), which requires a binary outcome and monotonicity. We have a continuous outcome and use the simple in-model fractional reduction 1 − Q_cf/Q_factual as a usable proxy. It tells you "what fraction of the model's predicted outcome would go away if this well had been shut off", which is informative but isn't a probability in the formal Bayesian-hypothetical sense.
  • Negative contributions are real and informative. For some wells the model predicts that shutting them off would increase nearby expected ML. This is usually because the well is in a formation/depth/fault context where its observed injection is correlated with lower-than-average outcomes — the model sees that pattern and the counterfactual breaks it. Don't dismiss negative contributions; they're a legitimate signal that the well isn't a primary driver under the model. They get a 0% share (we only divide positive contributions across the positive total).

5. How to read the per-well numbers

Number Interpretation
Q_factual The model's expected max ML at this well's actual feature vector on the event date. (Note: the "max ML" is the max within R km of the well, so a high Q_factual means "the model expects this well's neighborhood to have higher seismic outcomes given its feature profile" — not "this well's injection caused the event.")
Q_counterfactual The same prediction but with this well's cum_vol_365d set to 0 and BHP set to depth-only hydrostatic.
contribution Q_factual − Q_counterfactual. The model's estimate of how much of the predicted local ML goes away if you shut this well off. Can be negative.
share This well's positive contribution as a fraction of the sum of all positive contributions across the wells in the radius. Wells with negative contribution get 0%.
PN proxy Fractional reduction: 1 − Q_cf / Q_factual, clamped to [0, 1]. ≥0.5 = "more than half of the model's predicted outcome at this well would go away under shut-off." ≥0.7 = strong attribution under the model.

6. Known limitations

  • The Q is fit per-radius. The dashboard uses 20 different Q models depending on the radius slider. The same well at the same date can have a different attribution depending on the radius, because the outcome definition (max ML within R km) changes.
  • Wells outside the radius are invisible to the attribution. A well 8 km from the event with a 7 km radius contributes 0 to the in-radius decomposition, even if it's a plausible driver. Try the radius slider to see how the picture changes.
  • Multi-event days are aggregated by max ML. If two events happen on the same day at different locations, the panel records only the max nearby ML per well-day. Per-event attribution at single- event days is cleanest; multi-event days should be interpreted with extra caution.
  • Sum of contributions ≠ event magnitude. The contributions are per-well predictions from a population-fit model, not a decomposition of the event's actual ML into well shares. Don't expect Σ contribution_i to equal the observed M.
  • Leave-one-out vs Shapley. Our g-computation is "remove this one well, holding all other wells at their actual values". If there are strong interactions between wells (one well's pressure plume amplified by another's), Shapley values would give a different and more rigorous attribution. We'd need to fit Q across all subsets of nearby wells to compute true Shapley; not done.
  • Negative contributions mean what they say. If a well has contribution < 0, the model thinks shutting it off would NOT reduce expected nearby ML. This is informative — it tells the regulator that for this well, the shut-off lever wouldn't work according to the model. Don't ignore the sign.
